## ADDED Requirements

### Requirement: 取得最新匯率
系統應（SHALL）透過免費、免金鑰的公開匯率 API，自動取得主要外幣相對於台幣（TWD）的每日即時匯率。
取得後，系統應將匯率正規化為「1 外幣 = N 台幣」的正向格式後再儲存與計算（API 回傳以 TWD 為基準時，需先取倒數）。

#### Scenario: 成功取得 API 匯率
- **WHEN** 應用程式載入且網路連線正常時
- **THEN** 系統應發送 HTTP GET 請求至匯率 API，將回傳值正規化為正向匯率後更新匯率表

### Requirement: 多來源降級鏈
系統應（SHALL）以由上而下的降級順序取得匯率，任一層失敗即自動嘗試下一層，確保在任何情況下都能取得可用匯率：
1. 主來源 API（`https://open.er-api.com/v6/latest/TWD`）
2. 備援來源 API（jsDelivr CDN 上的 `@fawazahmed0/currency-api`）
3. `localStorage` 內的本地快取
4. 內建的寫死預設匯率

#### Scenario: 主來源失敗時改用備援來源
- **WHEN** 主來源 API 請求失敗（逾時、非 2xx 回應或解析錯誤）但網路仍可用時
- **THEN** 系統應改向備援來源 API 發送請求，成功後正規化並更新匯率表

#### Scenario: 所有線上來源皆失敗
- **WHEN** 主來源與備援來源皆無法取得匯率，但 `localStorage` 存有先前快取時
- **THEN** 系統應使用本地快取的匯率，並於 UI 顯示最後更新時間與「匯率可能過時」提示

#### Scenario: 首次離線且無任何快取
- **WHEN** 應用程式首次於離線狀態載入，且無線上來源亦無本地快取可用時
- **THEN** 系統應退回使用內建預設匯率，並於 UI 明確標示「此為預設估計值，請手動校正」

### Requirement: 請求節流與禮貌性存取
系統應（SHALL）以最低必要的頻率存取外部匯率來源，避免被任一來源視為濫用或自動化爬取。

#### Scenario: 快取仍在效期內不重複請求
- **WHEN** 應用程式載入且本地快取的 `lastUpdated` 距今未滿 24 小時
- **THEN** 系統不應（SHALL NOT）發送任何匯率 API 請求，直接使用本地快取的匯率

#### Scenario: 計算操作不觸發網路請求
- **WHEN** 使用者進行計算、新增商品或調整輸入等互動操作時
- **THEN** 系統應僅使用記憶體中既有的匯率，不應（SHALL NOT）為此發送任何匯率 API 請求

#### Scenario: 單一來源失敗的有限重試
- **WHEN** 某一線上來源請求失敗時
- **THEN** 系統至多重試一次，仍失敗則降級至下一來源，不應（SHALL NOT）對同一來源進行密集或無限的重試

### Requirement: 本地儲存快取
系統應（SHALL）將取得的匯率與時間戳記快取至瀏覽器的 `localStorage` 中。

#### Scenario: 離線時讀取快取
- **WHEN** 應用程式在無網路連線下載入，或 API 請求失敗時
- **THEN** 系統應從 `localStorage` 載入快取的匯率，並顯示該匯率最後更新的時間

#### Scenario: 快取過期檢查
- **WHEN** 應用程式載入且快取的匯率已超過 24 小時
- **THEN** 系統應重新嘗試發送 API 請求以更新匯率

### Requirement: 手動調整匯率與鎖定
系統應（SHALL）允許使用者手動輸入或微調任何貨幣的匯率。手動輸入的匯率必須為大於零的有限正數，否則應拒絕並提示。
一旦使用者手動覆寫某貨幣匯率，該貨幣即標記為「已鎖定」；每日自動更新流程應跳過已鎖定的貨幣、不予覆蓋，直到使用者主動解除鎖定。

#### Scenario: 手動覆寫並鎖定匯率
- **WHEN** 使用者手動為某個貨幣輸入自訂匯率（例如設定 1 USD = 33.1 TWD）
- **THEN** 系統應立即使用該新匯率進行後續計算，並將此覆寫值與「已鎖定」標記一併儲存於 `localStorage`

#### Scenario: 自動更新跳過已鎖定貨幣
- **WHEN** 每日自動更新匯率時，某貨幣已被使用者手動鎖定
- **THEN** 系統應保留該貨幣的手動匯率，僅更新其他未鎖定的貨幣

#### Scenario: 解除鎖定後恢復自動更新
- **WHEN** 使用者主動解除某貨幣的鎖定
- **THEN** 系統應於下一次更新時恢復以自動取得的匯率覆寫該貨幣

#### Scenario: 拒絕無效的手動匯率
- **WHEN** 使用者於手動匯率欄位輸入非正數（負值、0、空白或非數字）
- **THEN** 系統應拒絕該輸入並顯示錯誤提示，維持原匯率不變

### Requirement: 鎖定狀態的視覺呈現與控制項
系統應（SHALL）在介面上清楚呈現每個貨幣的鎖定狀態，並提供解除鎖定的控制項，使鎖定機制對使用者可見、可操作。

#### Scenario: 顯示已鎖定的貨幣
- **WHEN** 某貨幣因手動覆寫而處於「已鎖定」狀態
- **THEN** 系統應在該貨幣旁顯示明確的鎖定標示（如鎖頭圖示或「已鎖定」文字）

#### Scenario: 提供解除鎖定控制項
- **WHEN** 使用者點選已鎖定貨幣旁的「解除鎖定」控制項
- **THEN** 系統應移除該貨幣的鎖定標記，使其於下一次自動更新時恢復為自動取得的匯率

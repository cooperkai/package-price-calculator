## 背景與上下文 (Context)

使用者在購物時需要一個工具，能將不同包裝容量與外幣計價的商品，統一換算成**每 100g 的台幣（TWD）價格**以進行比價。
本應用程式將作為靜態網站代管於 **GitHub Pages**，這代表沒有後端伺服器。所有的計算、資料獲取與本地資料持久化都必須完全在瀏覽器端（用戶端）進行。
由於購物者在大型超市或地下室時經常遇到收訊不良或沒有網路的情形，本應用程式必須透過漸進式網頁應用程式（PWA）架構，具備強健的**離線優先（Offline-First）**能力。

## 目標與非目標 (Goals / Non-Goals)

**目標：**
- 提供一個乾淨、質感高級、現代化且行動裝置友好的單頁應用程式（SPA）。
- 自動取得當天最新匯率，且不要求 API 金鑰或進行註冊。
- 提供本地快取機制，使應用程式在離線時仍能完全正常運作。
- 支援多商品橫向比價，並提供直觀的「最划算」（每 100g 最便宜）視覺標示。
- 靜態託管於 GitHub Pages。

**非目標：**
- 建立後端伺服器、使用者帳號系統或登入機制。
- 將比價歷史儲存於雲端資料庫（使用 `localStorage` 已足夠）。
- 提供分秒即時的匯率波動追蹤（每日更新一次已足夠）。

## 設計決策 (Decisions)

### 1. 靜態網頁技術棧：原生 HTML、CSS 與 JS（無須建置步驟）
- **決策依據**：為最大化與 GitHub Pages 的相容性並保持極速的載入時間，我們將採用原生的 HTML5、CSS3 與現代 JavaScript (ES6)。不使用複雜的建置工具（如 Webpack 或 Vite），使儲存庫結構維持簡單、好讀且能直接發布。
- **替代方案評估**：曾考慮使用 Next.js 或 React。雖然 React 能使狀態管理更具結構化，但它會增加編譯負擔、打包體積，並可能在慢速行動網路上造成載入延遲。原生 JS 足以應付本專案需求，且能將載入時間控制在 100 毫秒以內。

### 2. 介面與樣式：原生 CSS 搭配自訂屬性（深色主題與毛玻璃效果）
- **決策依據**：使用 CSS 變數來管理色彩、間距與過渡動畫時間。我們將實作一個響應式的現代深色主題控制面板。卡片元件將採用細微的背景模糊（`backdrop-filter`）與細邊框，營造高級的「毛玻璃」（Glassmorphism）質感。配合 CSS Grid 和 Flexbox 佈局，確保在行動裝置上有著極佳的單手操作體驗。
- **字型**：採用系統字型堆疊（如 `-apple-system, "Segoe UI", Roboto, "Noto Sans TC", sans-serif`），不載入任何網路字型。如此可零下載、離線天生可用，並在各平台呈現原生的高級觀感。

### 3. 匯率資料：多來源降級鏈（Fallback Chain）

- **決策依據**：與其依賴單一 API（任一服務超出額度、改版或停機都會導致功能中斷），改採由上而下的多來源降級策略，任一層失敗就自動往下一層取值，確保 App 在任何情況下都能取得可用匯率。
- **降級順序**：
  1. **主來源** — ExchangeRate-API 開放端點：`https://open.er-api.com/v6/latest/TWD`。免金鑰、以 TWD 為基準。
  2. **備援來源** — fawazahmed0 currency-api（jsDelivr CDN 託管的純 JSON 檔）：`https://cdn.jsdelivr.net/npm/@fawazahmed0/currency-api@latest/v1/currencies/twd.json`。本質為 CDN 上的靜態檔，高可用、無流量限制焦慮，適合純前端靜態站。
  3. **本地快取** — 前兩者皆失敗（例如離線）時，讀取 `localStorage` 內上次成功取得的匯率，並於 UI 顯示「最後更新時間」與「匯率可能過時」提示。
  4. **寫死預設值** — 連快取都不存在（首次離線載入）時，退回內建預設匯率（例如 USD=32.5, JPY=0.21, EUR=35.0），並明確於 UI 標示「此為預設估計值，請手動校正」。
- **匯率方向正規化**：兩個 API 來源的基準皆為 TWD，回傳的是「1 TWD = N 外幣」（例：USD≈0.0308），方向與計算所需相反。系統取得後一律轉換成「**1 外幣 = N 台幣**」的正向格式（`N = 1 / 原始值`）後再儲存與計算，使其與計算公式、手動覆寫 UI 的方向一致。詳見 `unit-price-calculator` 規格。
- **禮貌性存取（避免被視為濫用 / 爬蟲）**：
  - **每日至多請求一次**：依快取的 `lastUpdated` 時間戳記判斷，未滿 24 小時直接使用快取，不重複發送請求。
  - **僅於 App 載入時檢查一次**：計算、新增商品等操作一律使用記憶體中的匯率，絕不在每次互動時打 API。
  - **失敗有限重試**：單一來源失敗最多重試一次後即降級至下一來源，不做無限輪詢或密集重試。
  - **善用 CDN 快取**：備援來源使用 jsDelivr `@latest`，請求會落在 CDN 邊緣節點，對來源伺服器幾乎零負擔。
  - 整體而言單一使用者單一裝置每日請求量極低（通常 0~1 次），不會觸發任何來源的速率限制。
- **手動覆寫與鎖定**：使用者可在介面上手動修改任一貨幣匯率；一旦手動覆寫，該貨幣即標記為「已鎖定」，每日自動更新流程會**跳過**已鎖定的貨幣、不予覆蓋，直到使用者主動解除鎖定。未鎖定的貨幣仍照常每日自動更新（詳見 `exchange-rate-service` 規格的「手動調整匯率與鎖定」需求）。

### 4. 資料儲存與離線化：LocalStorage 與 Service Worker
- **決策依據**：
  - `localStorage`：用以持久化儲存商品比價列表與快取的匯率資料（附帶 `lastUpdated` 時間戳記）。
  - `sw.js`（Service Worker）：對靜態資源（`css/style.css`、`js/app.js` 等）以「快取優先（Cache-First）」策略快取，確保離線時瞬間載入。
  - **快取版本管理**：靜態資源快取以版本號命名（如 `price-calc-v1`），每次部署新版即遞增版本號；Service Worker `activate` 事件會清除所有舊版號快取。`index.html` 改採「網路優先（Network-First）」策略，使用者一連上網即可取得最新版本，避免被舊快取卡住。

### 5. 測試策略：Vitest（單元・測試先行 TDD）＋ Playwright（E2E／UI）
- **核心邏輯模組化**：將計算、單位換算、匯率正規化、輸入驗證、分組與排名、降級鏈決策等純邏輯，抽成可被 import 的純函式模組（如 `js/calc.js`、`js/exchange-rate.js`），與 DOM 操作分離，才能在 Node 環境被測試。
- **單元測試（Vitest，測試先行）**：純邏輯採 TDD。由於各規格的 Scenario 已含明確輸入與預期輸出，流程為——先將 Scenario 逐條轉成 Vitest 測試（紅燈）→ 實作至全部通過（綠燈）→ 在綠燈保護下重構。可用 jsdom 與 mock 模擬 `localStorage`、`fetch`，以測試快取、降級與鎖定邏輯。
- **E2E／UI 測試（Playwright）**：難以單元測試的瀏覽器層行為交給 Playwright——離線載入（Service Worker）、響應式版面、PWA 安裝，以及完整使用者流程（輸入→計算→加入列表→最划算高亮）。
- **測試檔放置慣例（co-location）**：測試檔與被測模組放同一層、同名 `.test.js`（如 `js/calc.js` 配 `js/calc.test.js`）。此為 Vitest／Jest 生態的主流慣例，好處是「一眼看出模組有無測試、改名搬移一起帶走」；JS 不像 Java（Maven `src/test`）強制分離，Vitest 以 glob（`**/*.test.js`）尋找測試，放置位置不影響執行。
- **與「無建置部署」並存**：Vitest、Playwright 與 `node_modules` 皆為**開發期 devDependencies**，僅於本機／CI 執行，**不會被打包進部署的靜態網站**；GitHub Pages 上線內容仍是純 HTML/CSS/JS、零建置（`.test.js` 雖隨 repo 發佈但不會被 `index.html` 載入）。

## 風險與權衡 (Risks / Trade-offs)

- **[風險] 匯率 API 超出額度或服務中斷**
  - *緩解措施*：本程式會在本地快取匯率。若 API 無法連線，將繼續使用快取資料。若無快取資料，則退回預設值並在介面上提示使用者，允許使用者手動輸入調整。
- **[風險] 儲存空間上限與瀏覽器資料清理**
  - *緩解措施*：由於瀏覽器或使用者可能會清理 `localStorage`，比價資料是暫時性的。但對於一個快速購物的輔助工具來說，長期保存歷史紀錄並非核心關鍵需求。
- **[風險] 毫升（ml）與克（g）單位的近似換算**
  - *緩解措施*：為簡化計算，本程式預設液體密度為 1g/ml（適用於水、牛奶、果汁或醬料等大部分日常食品）。介面上將對此近似值進行提示說明。
